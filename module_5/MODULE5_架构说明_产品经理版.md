# Module 5 架构说明（产品经理版）

本文档面向**非技术背景**的同事（产品、运营、项目管理等），用**白话**说明 Module 5 在系统里做什么、数据怎么流、当前设计好在哪、以及技术同事提到的「改进点」分别**对你意味着什么**。文中尽量不写代码；偶尔出现的英文是数据库表名或配置名，方便你和研发对齐时指称。

---

## 1. 这份文档解决什么问题？

你可能听过「Module 5 是触达建议 / 微信草稿生成」之类的说法，但不清楚：

- 它**依赖**前面哪些模块？
- 产出**存在哪里**、**长什么样**？
- 技术评审里说的「架构」「run_id」「按需拉记忆」**对业务有什么影响**？

读完第一节到第四节，你能**向别人讲清楚 M5 在流水线中的位置**；读完第五节，你能理解**为什么要改某些东西、改了之后对演示和落地有什么好处**。

---

## 2. Module 5 一句话是干什么的？

**Module 5 的角色**：在已经有「趋势短名单」和「客户记忆」的前提下，为每一位（或你选中的）客户，自动生成：

- **触达角度**（从什么话题切入、为什么选这个角度）；
- **微信私聊草稿**（偏异步聊天开场，不是店里当面话术）；
- 以及系统用于**追溯、评估**的信息（用了哪些趋势、和记忆怎么对应等）。

可以把它理解成：**给 CA（客户顾问）的「写消息助手」**，输入是系统里已经结构化好的数据，输出是可编辑的草稿和建议。

---

## 3. 在整个产品流水线里，它站在哪里？

用一个**极简流程**说明（前面模块的细节你可以单独要别的文档）：

1. **更早的环节**会从社媒、业务数据里整理「趋势」相关结论，并筛出一份**趋势短名单**（可以理解为：这一批里最值得 CA 参考的趋势 TOP 列表）。
2. **Module 4** 会把和某位客户相关的笔记、对话等，整理成**结构化的客户记忆**（例如偏好、近况、下一步意向等字段）。
3. **Module 5** 同时读取：
   - 这份**趋势短名单**（来自 Module 2 写入数据库的结果）；
   - **客户记忆**（来自 Module 4 写入数据库的结果）。
4. **Module 5** 调用大模型，针对**每一位客户**生成触达建议与草稿。
5. 结果会：
   - 保存在本地的一个日志文件里（方便演示、导出、写评测报告）；
   - 在配置了数据库的情况下，写入 **Supabase** 里的两张表（方便和整条链路、以后的前台或报表对齐）。

**对你重要的结论**：M5 **不是**从零爬数据；它**强依赖** M2 的短名单和 M4 的记忆已经进库且质量可靠。若前面批次错了或 run 对不齐，M5 输出也会跟着偏。

## 4. 数据从哪来、到哪去？（对照表）

下面这张表方便你和研发、数据同学**对口径**。

| 方向 | 名称（可当作「仓库货架」） | 谁往里放数据 | Module 5 怎么用 |
|------|---------------------------|--------------|-----------------|
| 输入 | 趋势短名单表（`module2_trend_shortlist`） | Module 2 | 作为「只读知识库」：告诉模型这一批有哪些趋势可参考、各趋势要点是什么 |
| 输入 | 客户记忆表（`module4_client_memories`） | Module 4 | 先加载一份**客户列表（轻量信息）**；真正生成某位客户时，再**单独拉该客户完整记忆**（避免一次性把所有人的细节都塞进模型，费钱且乱） |
| 输出 | 触达建议表（`module5_outreach_suggestions`） | Module 5 写入 | 每位客户一条：角度、微信草稿、理由、用了哪些依据等 |
| 输出 | 运行批次表（`module5_run_logs`） | Module 5 写入 | 一次批量生成可对应一条「批次摘要」，便于排查和统计 |
| 输出 | 本地 `run_log.json`（文件） | Module 5 写入 | 演示、导出表格、飞书/谷歌表、质量评测脚本等常用这份 |

更正式的字段说明在技术文档 **`IO_CONTRACT.md`** 里；你需要「和 LVMH / 合规对字段」时可以请研发打开那份一起过。

## 5. 当前设计里，对产品/演示「已经友好」的地方

下面每一项都说明**它是什么**以及**你为什么该在意**。

### 5.1 「客户列表轻量 + 生成时再拉全量记忆」

- **是什么**：界面上或脚本里先只看到客户 ID、姓名、标签等列表信息；只有你（或系统）选中某位客户要生成时，才去数据库把这位客户的**完整记忆**取出来送给模型。
- **为什么好**：
  - **成本**：大模型按上下文长度计费，所有人全量一起塞进去会非常贵且容易超长。
  - **体验**：CA 通常一次只关心「这几位客户」，行为上更自然。
- **产品话术**：可以说我们支持**大规模客户池**，但**按人按需计算**，适合门店真实工作流。

### 5.2 「趋势知识库」可以瘦身（紧凑模式）

- **是什么**：趋势短名单里可能有很多字段（证据、指标等）。系统可以把送进模型的那一块做成**精简版**，只保留对写微信更有用的信息，减少噪音。
- **为什么好**：模型更稳定、更聚焦，也更省 token。
- **产品话术**：同一批趋势，**面向「写消息」场景做了专门裁剪**，不是把原始数据表直接贴给 AI。

**它「长什么样」（给非技术同事的直观理解）**  
你可以把「趋势知识库」理解成：**同一份趋势短名单的两种「打印版」**——数据还是数据库里那份，但在送给 AI 之前，会先经过一个叫 `trend_kb` 的整理步骤：

| 概念 | 白话 |
|------|------|
| 数据来源 | Module 2 写入库的**趋势短名单**（一批里有哪些 trend、分数、摘要等） |
| `compact`（默认） | **摘要版**：每条趋势只保留标签、摘要、部分证据、关键指标等，长文截断，减少废话 |
| `full` | **完整版**：几乎把短名单里能带的字段都带上（在 Top-N 之后），适合调试或要强证据时 |
| 是不是 RAG？ | **不是**。这里没有「先检索再拼上下文」的向量检索；只是**同批数据的裁剪/不裁剪**。真正的检索型 RAG 用在下面的**商品目录**（5.3）。 |

环境变量 **`M5_TREND_KB_MODE`** 在 `compact` 和 `full` 之间切换（见根目录 `.env.example`）。

### 5.3 商品目录：RAG（检索增强生成）

- **是什么**：Module 1 的 `module1_brand_products` 里有很多 SKU。生成某位客户的微信草稿时，系统会用 **这位客户的记忆 + 本批趋势知识库** 拼成一段「查询」，在商品库里做 **语义检索（向量相似度）**，只把**最相关的 Top-K 条**商品放进提示里，而不是整本目录贴进去。
- **是不是 RAG？** **是。** 标准说法叫 **RAG（Retrieval-Augmented Generation）**：先 **R**etrieval（检索出相关片段），再 **A**ugment（把片段塞进给模型的上下文），最后 **G**eneration（模型生成 outreach）。若嵌入接口失败，会**自动退回到简单的词重叠打分**，保证功能不中断。
- **为什么好**：目录变大以后，比「全表粘贴」**更省成本、更聚焦**，且仍能给模型**可引用的真实 SKU**（名称、链接、价格等以库为准）。
- **产品话术**：我们按**客户 + 当前趋势上下文**「智能翻产品目录」，只把**可能用得上的款**交给 AI 写草稿。

可调参数见根目录 `.env.example`（`M5_CATALOG_RAG`、`M5_CATALOG_RAG_TOP_K`、`M5_CATALOG_EMBED_MODEL` 等）。若将 `M5_CATALOG_RAG=0`，则回退为「截断后的列表粘贴」（便于对照实验）。

### 5.4 输入默认只从数据库来

- **是什么**：配置好数据库后，趋势和客户记忆都从 Supabase 读，避免演示现场「忘带本地 JSON」或版本不一致。
- **为什么好**：**演示和真实环境一致**，适合对外讲「端到端流水线」。
- **注意**：若数据库没配好，Module 5 会直接报错提示配置问题——这不是 bug，而是**强制统一数据源**的设计选择。对 PM 的意义是：**彩排时要确认 .env / 数据库里有最新一批 M2、M4 数据**。

### 5.5 周边工具齐全（对汇报和迭代有用）

- **终端 / 命令行**：适合技术彩排、批量跑。
- **简易网页（本机 CA 工作台）**：勾选客户再生成，更像「给业务试用」的形态（需在安全环境下使用，一般仅本机）。
- **评测脚本**：用另一套「质量审计」提示词给草稿打分，方便迭代提示词和模型。
- **导出**：可把结果铺到表格或飞书，方便你写周会材料或客户共创。

这些说明团队已经在按**「能演示、能测、能交付」**的路径建设，而不止是一个孤立脚本。

## 6. 技术同事提到的「建议」——逐项用白话讲透

下面每一条都按同一结构写：**是什么 → 对你（产品/业务）的影响 → 建议动作（通常要研发做）→ 你不做会怎样**。

---

### 建议 A：统一「一次批量生成」的批次编号（run_id）

**是什么**  
系统里有一个叫 **run_id** 的东西，用来表示「这是哪一次流水线/哪一批生成」。理想情况是：**同一次操作里生成的所有客户，共用同一个批次号**，这样数据库里的记录、本地日志、以后报表都能对得上。

当前实现里存在一个**细节不一致**：  
- 写在电脑上的 `run_log.json` 里，**每一条客户记录**可能各自带了一个按「生成时刻」生成的时间戳式编号；  
- 写入数据库时，**整批**又会用**另一个**统一的编号。

**对你（产品）的影响**

- **追溯困难**：若你要回答「4 月 10 日下午那一场演示，一共生成了多少人、数据库里对应哪一批？」，研发和数据分析可能要**多对几次**才能把文件和表对齐。
- **报表/看板**：若未来做「按批次看通过率、平均分、采纳率」，批次号不统一会增加**统计口径**争议。
- **合规/审计**：若需要证明「某次输出的输入快照是什么」，批次对齐越清晰越好。

**建议动作**  
由研发在**开始跑这一批客户之前**就生成**一个**批次号，**同一批里所有客户**、**本地文件**、**数据库**都用它。

**若暂时不改**  
功能仍可用，演示也能做，但**跨系统对账、事后分析**会略痛苦，时间越长越明显。

---

### 建议 B：`agent.py` 文件职责过多，将来要「拆小文件」

**是什么**  
研发把很多逻辑写在同一个大文件里：连模型、解析返回格式、命令行交互、打印结果、写文件、同步数据库等都在一起。这在**早期 MVP 很快**，但文件会像「万能遥控器」——改一个功能容易碰到别的功能。

**对你（产品）的影响**

- **迭代速度**：当你要加「流式输出、重试策略、换模型、A/B 两套提示词」时，研发**改起来更慢、更容易引入新 bug**。
- **分工**：以后若不止一人维护，**协作冲突**会变多。
- **风险**：**你不直接改代码**，但会感知为「同样一个需求，以前两天现在一周」。

**建议动作**  
不要求你定技术方案；你可以在产品排期里接受「**内部重构 0.5～1 个迭代**」这类任务，目标只有一句话：**让 M5 核心逻辑更好维护**，不对外改变功能行为（或仅极小变更）。

**若暂时不改**  
短期没问题；需求复杂度上来后，**研发会频繁喊「得先重构」**。

---

### 建议 C：导入模块的方式有点「隐式」，宜改成更正规的包结构

**是什么**  
研发提到：写数据库的那段代码，有时通过「Python 路径」碰巧能找到文件，读起来不够直观，新人容易误以为和别的模块的同名文件是一套。

**对你（产品）的影响**  
几乎**没有直接的产品功能差异**；主要是**工程可靠性和新人上手速度**。若团队扩大或外包参与，**踩坑概率**会高一点。

**建议动作**  
列为**低优先级、纯技术债**；你可以在复盘里记一笔，不必对外承诺。

**若暂时不改**  
对当前演示和单维护者场景**影响很小**。

---

### 建议 D：顶部说明文字与「只能从数据库读」的现状不一致

**是什么**  
脚本最上面的注释或 README 里，可能还留着「以前可以从本地 JSON 读」之类描述，但程序已经改成**没有数据库配置就会直接报错**。

**对你（产品）的影响**  
**新同事、外部伙伴**按旧文档操作会困惑；**彩排浪费时间**在「为什么报错了」上。

**建议动作**  
请研发或文档负责人**改注释与对外演示 checklist**（半日到一天量级），与真实行为一致。

**若暂时不改**  
**沟通成本**和**彩排摩擦**会反复出现。

---

### 建议 E：环境变量加载在多个文件里各写一份

**是什么**  
好几个脚本各自「读一遍 .env」。功能上没问题，但以后若改「从哪读配置、优先级怎么定」，要改多处。

**对你（产品）的影响**  
极小；偶尔表现为「**为什么 A 脚本能跑、B 脚本读不到 key**」这类**环境差异**。

**建议动作**  
技术债，**随重构一起做**即可。

---

### 建议 F：评测（eval）主要读本地文件，与「以数据库为准」的输入策略略不对称

**是什么**  
质量评测脚本通常读本地的 `run_log.json`。而正式跑批时，输入已经以数据库里的 M2/M4 为准。

**对你（产品）的影响**  
- 若你的流程是：**先跑 M5 → 立刻用 eval 脚本打分**，则**没问题**。  
- 若未来希望：**只从数据库里抽历史批次做回归评测**，和现在脚本习惯可能不一致，需要**多一步导出或改脚本**。

**建议动作**  
当你规划「**线上回归、每月质量看板**」时，和研发对齐：**评测数据源**是「本地文件」还是「数据库里的 run 日志字段」，写进 PRD 的「数据与验收」一节即可。

**若暂时不改**  
不影响当前**手工演示 + 本地评测**。

---

## 7. 和对外讲解时，你可以直接用的「三句话版本」

1. **Module 5 在 M2 趋势短名单和 M4 客户记忆都进库之后**，为选定客户生成**微信触达角度和草稿**，并写入数据库和本地日志，方便演示和评测。  
2. **设计上按真实 CA 工作流**：先看客户池，再**按人拉全量记忆**；趋势侧是**只读、可裁剪的知识库**；商品目录用 **RAG Top-K** 控制进模型的 SKU 数量。  
3. **近期最值得排进迭代的改进**是：**统一一批生成的批次号**，让日志、数据库、未来报表**口径一致**；其余多为**可维护性与文档一致性**，随团队规模再排期。
